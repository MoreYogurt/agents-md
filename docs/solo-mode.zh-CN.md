# 单人模式 — 面向单人项目的规范

> English version: [`solo-mode.md`](./solo-mode.md)

主规范（[`STANDARDS.zh-CN.md`](../STANDARDS.zh-CN.md)）假设的是团队环境：代码评审、分支保护、CI/CD、预览部署、云数据库。但相当一部分 AI 原生开发发生在完全不同的地方——**一个产品负责人 + 一个 AI 结对**，做本地优先的应用，没有流水线，也没有评审者。

单人模式是同一套纪律在这种环境下的重新推导：凡是关于 *AI 行为* 的规则原样保留，凡是关于 *协调多个人* 的规则替换成单人等价物。

假设环境：

- 一个人（产品决策、验收测试）+ AI（全部实现）。
- 本地优先或纯客户端产品（移动 App、桌面工具、CLI），数据存在用户设备上。
- 无 CI/CD、无 staging、无分支保护，项目初期甚至可能没有远端仓库。

## 十条规则

### 1. 修改可以频繁，提交必须有形

AI 改代码不限次数，但一个 commit 必须是功能的最小**可独立验证**切片——能跑、能渲染、或能独立通过测试。禁止 `wip` / `update` / `临时保存` 式提交；禁止把不相关改动塞进同一个 commit。

团队里这条规则保护的是评审者；单人项目里它保护的是你唯一拥有的历史：两周后出问题时，对可验证切片做 `git bisect` 就是你的代码评审。

### 2. 默认直接在 `main` 上工作

特性分支的存在意义是串行化评审、并行化人力。只有一个人、没有评审关卡时，分支是纯开销。小步、验证过的切片直接提交 `main`。

只有两种情况开分支，命名 `exp/<主题>`：

- 会让代码树坏掉一阵子的大范围重构；
- 真的可能整个丢弃的实验方案。

答案一旦明确就立即合回或删除。活过一周的 `exp/` 分支，本质是一个你在回避的决定。

### 3. 一个功能 = 一串 commit + 一次进度更新

团队用 PR 划定功能边界；单人项目的边界放在进度文件里（如 `PROGRESS.md`）：功能的 commit 落地后勾掉任务、记下遗留。发布节点用 `main` 上的附注 tag（如 `phase-1`）——tag 就是你的发布分支。

### 4. 没有 CI，提交前门槛就是 CI

CI 流水线能拦住的东西必须在每次 commit 前拦住：类型检查、lint、已有测试，UI 改动还要确认目标页面真的能渲染。**门槛不过，commit 不发生，没有例外。** 它是整个系统里唯一的质量关卡，破一次就名存实亡。

### 5. 只从 `main` 上打过 tag 的提交出包

任何离开你机器的东西——应用商店构建、TestFlight 上传、发给朋友的安装包——都从 `main` 上打过 tag 的 commit 构建，绝不从工作区的当前状态随手出包。"生产只从 `main` 部署"这条在单人场景下依然成立，只是少了部署机器人。

### 6. 本地数据库同样需要迁移纪律

没有云数据库不等于不需要迁移。schema 变更一律走版本化迁移脚本，禁止直接改表。每个迁移必须回答"老数据怎么办"——用户设备上的数据是唯一副本，迁移吃掉数据时没有运维团队、没有快照可恢复。

### 7. 聚合 schema 变更

先设计好功能的完整数据模型，再为它写**一个**迁移。十个"加一列"的迁移等于十次丢用户数据的机会，且没有任何附加价值。（与主规范一致——考虑到每个迁移都在终端用户设备上无人值守地运行，这条只会更重要。）

### 8. 先搜索，后生成

写任何新代码前，先搜项目里有没有现成实现：组件、工具函数、类型、i18n key、样式 token。**改现有文件优先于新建文件，复用优先于重写。** 平行实现是 AI 结对最昂贵的习惯，而单人项目没有评审者能发现第二份拷贝。（与主规范一致。）

### 9. 限制重试，诚实上报

同一问题失败 3 次就必须停：把现象、已试方案、当前判断写进进度文件的 Blockers 区，然后问人。禁止无限修复循环烧 token；禁止靠删掉失败的测试或 mock 掉报错来"解决"问题。（与主规范一致——单人场景下你上报的对象就是产品负责人本人，成本极低。）

### 10. 没有远端 = 没有备份

单人项目最大的风险不是合并冲突，是磁盘。仓库一建立就尽早配一个私有远端（免费的 GitHub 私有仓库、纯当备份用即可），至少也要在每个里程碑归档一次。并且**永远不要删除还没提交到任何地方的文件**——对未入库的工作而言，`rm` 是不可逆的。

## 相对主规范改了什么，为什么

| 主规范 | 单人模式 | 原因 |
| --- | --- | --- |
| `ai/<feature>` 分支 + PR + squash 合并 | 默认 `main`；仅重构/实验开 `exp/<主题>` | 没有评审者需要串行化，分支是纯开销 |
| 禁止 `on: push` 完整部署；PR 触发 CI | 规则 4：提交前门槛取代 CI | 没有流水线，关卡前移到 commit 时刻 |
| 迁移直连、运行时连接池、Neon 分支 | 规则 6–7：本地库版本化 + 聚合迁移 | 没有云数据库，但迁移纪律原样适用 |
| 事件驱动而非轮询 | 并入规则 9 | 单人场景下它的本质就是"别空转烧 token" |
| 先搜索后生成；限制重试 | 原样保留（规则 8–9） | AI 行为规则与团队规模无关 |
| —— | 新增规则 10（备份） | 主规范从 GitHub 免费获得备份；单人项目常常一开始连远端都没有 |

## 什么时候"毕业"回主规范

单人模式是起点，不是身份。第二个贡献者出现的那一刻——或者有了 CI 流水线、staging 环境——就把受影响的规则切回主规范。转换是机械的：规则 2 变回分支策略，规则 3 变回 PR，规则 4 变回 CI，规则 5 变回 main 部署。
